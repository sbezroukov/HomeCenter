# Дополнения к описанию сервиса ao-bestsend-participant-service в Wiki

## Новые API (описание в том же стиле)

### Health - Get  
**Проверка работоспособности сервиса.**

`GET https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/health`

**Параметры**  
Запрос без параметров.

**Ответ**  
- HTTP 200 — сервис доступен.  
- В теле при необходимости: `{"status":"healthy","version":"..."}`.

**Логика работы**  
Эндпоинт проверяет, что приложение запущено и способно обрабатывать запросы. Может использоваться для liveness/readiness в K8s и мониторинга.

---

### Keys / {bankId} - Get  
**Получение информации по ключу для одной точки.**

`GET https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/keys/{bankId}`

**Параметры**  
| Параметр | Тип   | Описание                |
|----------|--------|--------------------------|
| bankId   | string | Идентификатор точки (path) |

**Ответ**  
- `responseCode` — 0 успех, -1 ошибка (точка не найдена или ошибка БД).  
- Один объект `CertificateKeyDto` (как в Keys - Get) или пустой ответ при отсутствии точки.

**Логика работы**  
По переданному `bankId` выполняется запрос в БД к представлениям/таблицам ключей. Результат маппится в DTO и возвращается. Используется, когда нужны данные только по одной точке.

---

### Certificates / Validate - Post  
**Проверка наличия и валидности сертификатов по списку имён ключей.**

`POST https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/certificates/validate`

**Параметры**  
Тело запроса: массив имён ключей `KeyNames` (аналогично Certificates - Post).

**Ответ**  
- `responseCode` — 0 успех, -1 ошибка.  
- Массив объектов с полями: `KeyName`, `IsValid` (bool), `ErrorText` (при отсутствии или невалидности).

**Логика работы**  
Сервис по списку `KeyNames` проверяет наличие записей в БД и при необходимости валидность сертификата (срок действия и т.п.). Возвращает только статус проверки без самих данных сертификатов. Ограничение по количеству ключей — как в Certificates (например, MaxCountCertificates).

## Что ещё можно дописать в описание сервиса в Wiki

1. **Версионирование API**  
   Явно указать версию API (например, v1), политику обратной совместимости и план по введению v2 при необходимости.

2. **Лимиты и квоты**  
   Ограничения: MaxCountCertificates = 5, лимиты по RPS на ключевые эндпоинты (если есть), таймауты запросов.

3. **Формат ошибок**  
   Единый формат тела ошибки (HTTP 4xx/5xx): код, сообщение, correlationId, ссылка на документацию или тип ошибки (DB, Validation, NotFound).

4. **Примеры запросов/ответов**  
   Для каждого API — примеры curl или JSON (успешный ответ и типичные ошибки) для тестирования и интеграции.

5. **Мониторинг (расширить)**  
   Какие метрики планируются: количество запросов по эндпоинтам, латентность, ошибки, зависимость от БД. Куда отдаются метрики (Prometheus/Grafana и т.д.).

6. **Проверка работоспособности (детализация)**  
   Что именно проверяет health: только процесс, или также доступность БД, таймаут ожидания ответа. Рекомендации по использованию в K8s (liveness vs readiness).

7. **Схема и жизненный цикл сертификатов**  
   Краткое описание: откуда берутся данные (ТК, ЦС), как часто обновляются, кто отвечает за актуализацию в БД.

8. **Безопасность**  
   Требования к доступу к API (внутренняя сеть, mTLS, авторизация по токену/ролям), маскирование чувствительных данных в логах.

9. **Зависимости сервиса**  
   Явный список: БД (Contact, ContactDict), версии .NET, используемые библиотеки (если критичны для безопасности или совместимости).

10. **Чек-лист перед релизом**  
    Краткий чек-лист: обновление конфигов в Vault, проверка прав пользователя БД, smoke-тесты ключевых API после деплоя.

11. **Контакты и эскалация**  
    Контакты ответственных (уже есть), плюс: кто привлекается при инциденте (DevOps, DBA), ссылки на чаты/тикеты.

12. **Глоссарий**  
    Термины: точка, ключ, сертификат, прямой банк, CertCenter — краткие определения для новых сотрудников и смежных команд.
