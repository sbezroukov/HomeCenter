Дополнения к паспорту сервиса ao-bestsend-participant-service

Документ содержит описание новых API в формате, едином с существующей документацией сервиса (раздел «API сервиса»), а также рекомендации по расширению описания сервиса в Wiki.

API сервиса (новые эндпоинты)

Health – Get

Назначение: проверка работоспособности сервиса.

URL: GET https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/health

Параметры

Запрос без параметров. Тело запроса не передаётся.

Ответ

  • HTTP 200 — сервис доступен и готов обрабатывать запросы.
  • Тело ответа (опционально): JSON-объект с полями status (значение "healthy"), version (версия приложения при наличии).
  • При недоступности сервиса возвращается HTTP 503 или соединение не устанавливается.

Логика работы

При получении GET-запроса по пути /api/v1/Participant/health сервис проверяет, что процесс запущен и способен обрабатывать входящие запросы. Дополнительно может выполняться проверка доступности подключения к базе данных (в зависимости от реализации). Эндпоинт предназначен для использования в пробах жизнеспособности (liveness) и готовности (readiness) в Kubernetes, а также для мониторинга доступности со стороны балансировщика или системы мониторинга. Время ответа должно быть минимальным, без тяжёлых операций.

Обработка ошибок

При внутренней ошибке приложения возвращается HTTP 503 Service Unavailable. Детализация ошибки в теле ответа не обязательна для данного эндпоинта.

Keys / {bankId} – Get

Назначение: получение информации по ключу для одной точки (банка) по её идентификатору.

URL: GET https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/keys/{bankId}

Параметры

ПараметрРасположениеТипОписание
bankIdPath (путь URL)stringИдентификатор точки (банка). Обязательный. Передаётся в пути запроса, например: /api/v1/Participant/keys/12345.

Ответ

  • responseCode — 0 при успешном выполнении; -1 при ошибке (точка не найдена, ошибка доступа к БД или иная внутренняя ошибка).
  • При responseCode = 0: один объект типа CertificateKeyDto, структура которого совпадает с элементами массива keysInfo в ответе эндпоинта Keys – Get (BankId, UKey, PpCode, DirectBank, CertCenter, AllowedExternalIps, AllowedInternalIps, ExtIps, IntIps).
  • При responseCode = -1 или если точка с указанным bankId не найдена: пустой объект или отсутствие данных в теле ответа (в соответствии с принятой в сервисе схемой ответа об ошибке).

Логика работы

При получении запроса сервис извлекает значение bankId из пути URL и выполняет запрос к базе данных к тем же объектам, что и для эндпоинта Keys – Get (таблицы и представления: ContactDict.dbo.TCNT_KEYS, Contact.dbo.tCNT_Cert, Contact.dbo.tCNT_BANKS, Contact.dbo.tCNT_DIRECT_BANK), с фильтрацией по идентификатору точки. Полученная запись преобразуется в DTO CertificateKeyDto и возвращается в ответе. Эндпоинт предназначен для сценариев, когда потребителю нужны данные только по одной конкретной точке, без загрузки полного списка ключей.

Обработка ошибок

При ошибке подключения к БД или выполнении запроса в ответе возвращается responseCode = -1 и пустой или нулевой результат. При передаче несуществующего или некорректного bankId поведение должно быть согласовано с общим форматом ошибок сервиса (например, responseCode = -1 и отсутствие данных).

Certificates / Validate – Post

Назначение: проверка наличия и валидности сертификатов по списку имён ключей без возврата самих данных сертификатов.

URL: POST https://ao-bestsend-participant-service.testing.qiwi.com/api/v1/Participant/certificates/validate

Параметры

Тело запроса (application/json): объект с массивом имён ключей, например KeyNames — массив строк. Формат аналогичен телу запроса эндпоинта Certificates – Post.

ПараметрТипОписание
KeyNamesstring[]Массив имён ключей (наименований сертификатов), по которым требуется проверить наличие и валидность. Обязательное поле.

Ответ

  • responseCode — 0 при успешном выполнении запроса; -1 при внутренней ошибке (например, недоступность БД).
  • Массив объектов с полями: KeyName (string) — имя ключа; IsValid (bool) — признак наличия и валидности сертификата; ErrorText (string) — текст ошибки или причина невалидности (например, «Not_found», «Expired»). Заполняется при IsValid = false.
  • Количество элементов в массиве соответствует количеству переданных имён ключей (с учётом ограничения MaxCountCertificates, если применимо).

Логика работы

Сервис принимает список имён ключей KeyNames. Применяется то же ограничение на максимальное количество ключей в одном запросе, что и для эндпоинта Certificates – Post (на момент написания документации — MaxCountCertificates = 5): если в запросе передано больше ключей, обрабатываются только первые MaxCountCertificates. По ограниченному списку имён выполняется запрос к БД (Contact.dbo.tCNT_Cert и связанные объекты) для получения записей о сертификатах. Для каждой записи при необходимости проверяется валидность (например, срок действия). Результат формируется в виде массива с полями KeyName, IsValid, ErrorText. Сами данные сертификатов (поле Value) в ответ не включаются. Если сертификат по имени не найден, в соответствующем элементе массива выставляется IsValid = false, в ErrorText указывается, например, «Not_found».

Обработка ошибок

При ошибке доступа к БД или иной внутренней ошибке в ответе возвращается responseCode = -1 и пустой массив данных. При частичной ошибке (например, недоступность части данных) допустимо вернуть responseCode = 0 и заполнить ErrorText для соответствующих ключей. Рекомендуется документировать такие кейсы в описании API.

Рекомендации по дополнению описания сервиса в Wiki

Ниже перечислены разделы и формулировки, которые можно добавить или уточнить в паспорте сервиса в Wiki, чтобы описание было полнее и удобнее для эксплуатации и интеграции.

1. Версионирование API

Указать в паспорте текущую версию API (например, v1) и зафиксировать политику обратной совместимости: гарантируется ли сохранение формата запросов и ответов при обновлениях сервиса, в течение какого срока. При необходимости описать план перехода на v2 (сроки, поддержка старой версии).

2. Лимиты и квоты

Вынести в отдельный подраздел все ограничения: максимальное количество ключей в одном запросе сертификатов (MaxCountCertificates = 5), ограничения по RPS на ключевые эндпоинты (если заданы на уровне балансировщика или приложения), рекомендуемые таймауты запросов со стороны потребителей. Это снижает количество обращений в поддержку при интеграции.

3. Единый формат ошибок

Описать общий формат тела ответа при ошибках (HTTP 4xx/5xx): код ошибки, текстовое сообщение, опционально correlationId (или requestId) для поиска в логах, тип ошибки (DB, Validation, NotFound и т.д.). При наличии отдельной страницы с кодами ошибок — указать ссылку.

4. Примеры запросов и ответов

Для каждого описанного API добавить примеры: curl-запрос и примеры успешного JSON-ответа и типичных ошибочных ответов. Это ускоряет тестирование и интеграцию и уменьшает количество уточняющих вопросов.

5. Мониторинг

Расширить раздел «Мониторинг»: перечислить метрики (количество запросов по эндпоинтам, латентность, доля ошибок, доступность БД), способ сбора (Prometheus, иное) и место отображения (Grafana, Mission Control и т.д.). Указать, какие алерты настроены и на что они срабатывают.

6. Проверка работоспособности (детализация)

Уточнить, что именно проверяет эндпоинт health: только факт запуска процесса или также доступность БД, таймаут ожидания ответа. Рекомендации по использованию в Kubernetes: какой путь использовать для liveness, какой для readiness (если различаются), и как интерпретировать коды ответа.

7. Схема и жизненный цикл данных сертификатов

Кратко описать источник данных (таблицы/представления БД), откуда берутся сертификаты и ключи, как часто данные обновляются и кто отвечает за актуализацию (администраторы БД, отдельный процесс и т.д.). Это помогает при расследовании инцидентов и при планировании изменений.

8. Безопасность

Описать требования к доступу к API: только внутренняя сеть, необходимость mTLS, использование токенов или ролей. Указать, какие чувствительные данные (например, части сертификатов или строк подключения) не должны попадать в логи и как это обеспечивается.

9. Зависимости сервиса

Явно перечислить внешние зависимости: базы данных (Contact, ContactDict), версия .NET, критичные для безопасности или совместимости библиотеки. При изменении версий зависимостей обновлять этот раздел.

10. Чек-лист перед релизом

Добавить краткий чек-лист: проверка и при необходимости обновление конфигурации в Vault (в т.ч. CONNECTION_DB), проверка прав пользователя БД, выполнение smoke-тестов ключевых API после деплоя, проверка логов и метрик.

11. Контакты и эскалация при инцидентах

Помимо контактов разработчиков, указать, кого привлекать при инциденте (DevOps, DBA, владельцы БД), ссылки на чаты или системы заявок для эскалации.

12. Глоссарий

Добавить краткие определения терминов: точка (банк), ключ, сертификат, прямой банк (DirectBank), CertCenter и т.д. Это упрощает онбординг новых сотрудников и взаимодействие со смежными командами.
